# Golden path — the skill pack, end to end

This is a **worked reference**: the whole skill chain run on real fixtures to produce a complete
platform repo, so the team can see the entire build → catalogue → consume → go-live loop in one place
rather than as 13 separate demos. Everything here was generated by the skills' own scripts — nothing
hand-written except the small input specs in `inputs/`.

## Run it yourself

```bash
cd examples/golden-path
bash run_golden_path.sh          # macOS / Linux; pure Python, no database needed
```
(The BOP semantics step uses the real module at `10-workstreams/pb-migration/BOP Registration/...`; set
`PB_MODULE` / `PB_INVENTORY` to point elsewhere, or it's skipped.) The result is `platform-repo/`.

## What each step produced (the loop)

| Step | Skill | Output in `platform-repo/` |
|---|---|---|
| 1 | `repo-scaffold` | the mono-repo skeleton + conventions + `.pre-commit-config.yaml` + `tasks.py` |
| 2 | `onboard-source` | `ingestion/ird/bronze_ird.sql` — Bronze DDL, money widened to `Decimal(38,2)` |
| 3 | `import-schema-to-catalogue` | `dbt/models/_sources/ird__sources.yml`, `catalogue/import/ird_technical.yaml`, `catalogue/vocabularies/taxpayer_status.yml` |
| 4 | `legacy-module-to-openmetadata` | `catalogue/module-semantics/BOP_Registration_OpenMetadata.{yaml,md}` (25 tables / 141 cols, DRAFT) |
| 5 | `build-dbt-model` | `dbt/models/staging/stg_*`, `intermediate/int_taxpayer__master` (golden record + join contract), `marts/mart_debt__aged_balances` |
| 6 | `add-dq-checks` | `dbt/models/_dq/mart_debt__aged_balances.dq.yml` + singular tests + `quality/thresholds/...` |
| 7 | `build-superset-dashboard` / `expose-api` | `consumption/dashboards/debt_management/*`, `consumption/api/debt_*` |
| 8 | `onboard-consumer` | `consumption/consumers/sas_risk/{onboarding_plan.md,gap.json}` (SAS served by reuse) |
| 9 | `production-readiness-check` | `ops/readiness_report.md` — **GATE PASS** |

## What to look at first

- **`dbt/models/intermediate/int_taxpayer__master.sql` + `.yml`** — the Taxpayer-360 golden record:
  the join is a named model, and its key carries `not_null` + `relationships` tests (the join
  **contract**). The mart composes from it; it never re-joins raw sources.
- **`catalogue/module-semantics/BOP_Registration_OpenMetadata.yaml`** — what the legacy module's data
  meaning looks like recovered from source alone (DRAFT, ready for `verify-catalogue-semantics`).
- **`dbt/models/_dq/mart_debt__aged_balances.dq.yml`** — every test tagged by DQ dimension; plus the
  freshness/row-budget/accuracy singular tests.
- **`consumption/consumers/sas_risk/onboarding_plan.md`** — the second consumer onboarded mostly by
  reuse, with the write-back mart planned.
- **`ops/readiness_report.md`** — the go-live gate: auto-checks (CI, runbook, rollback, DQ, RBAC,
  alerts, secrets) + attestations, all green.

## The point

One spec per step, one command per step, deterministic output, gates that compose. Generate offline,
run `dbt build`/loads/imports on the stack, commit on the workstation. This `platform-repo/` is what a
Debt Management slice looks like once the pack has been run over it — the shape every later source,
mart and consumer follows.

> Note: `platform-repo/` is a committed **example**, not the team's live repo. The team scaffolds their
> own with `repo-scaffold` and runs the same chain on their real sources.
